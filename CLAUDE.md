# CLAUDE.md

Guidance for AI agents working in this repository. Keep changes consistent with the patterns below.

## What this is

`grbpwr-manager` (binary: `products-manager`) is the Go backend for the grbpwr.com store. It serves a
single process exposing **gRPC + a JSON/REST gateway (grpc-gateway) + Swagger UI** on one port (`:8081`),
backed by MySQL, with a set of background workers. It powers the public storefront, the admin panel, and
customer auth.

- Language: **Go 1.26**, module `github.com/jekabolt/grbpwr-manager`
- API contracts: Protobuf, compiled with **buf**
- Storage: MySQL (managed, via `sqlx`) + DigitalOcean Spaces (S3) for media
- Hosting: DigitalOcean App Platform (Docker image from `/Dockerfile`)

## Common commands

```bash
make run          # full build (proto + codegen + mocks) then `source .env && ./bin/products-manager`
make run-quick    # same but skips mockery (use on Go/mockery version mismatch)
make build        # format-proto → proto → generate → swagger → compile to bin/products-manager
make build-quick  # build without mockery
make proto        # buf format + buf generate (regenerate *.pb.go / *.gw.go)
make generate     # resend client + mocks + `go generate ./...`
make internal/statics  # rebuild combined Swagger JSON served by the API
make lint         # golangci-lint over ./internal/...
make cov          # tests with coverage (excludes /gen, /proto, /dependency)
make init         # clean + install tooling + proto + generate (first-time setup)
make image        # docker build grbpwr/grbpwr-pm:master
```

Local prerequisites: Go 1.26, `buf`, and the codegen tools from `make install` (protoc plugins, mockery,
oapi-codegen). Config/secrets for `make run` come from `.env` (gitignored).

Run a single test: `go test ./internal/<pkg>/... -run TestName`.

## Architecture

Entry: `cmd/main.go` (cobra root + `version` subcommand, `-c/--config` flag) → `cmd/run.go` → `app.App`.

`app/app.go` `App.Start` wires everything in order and is the map of the system:
1. `store.New` — MySQL connect + auto-migrate (`internal/store`, migrations in `internal/store/sql/`)
2. `mail.New` + worker — transactional email (**Resend**, despite the `sendgrid_*` config key names)
3. Background workers: `ordercleanup`, `storefrontcleanup`, `tiermanagement`, `stripereconcile`, `ga4sync`
4. `bucket.New` — DO Spaces media store
5. `auth.New` — admin JWT auth
6. `stripe.New` ×2 — live + test card processors (`StripePayment`, `StripePaymentTest`)
7. `revalidation` (Vercel ISR), `ga4`/`ga4mp`/`bigquery` (analytics)
8. Three gRPC services registered on the HTTP server: `admin`, `frontend` (storefront), `auth`

`App.Stop` stops workers **before** closing the DB (workers hold connections).

### Layers
- `proto/{admin,auth,frontend,common}/**.proto` → generated Go in `proto/gen/` (via `buf`, config `buf.gen.yaml`/`buf.work.yaml`). **Never edit generated `*.pb.go` / `*.gw.go`** — change the `.proto` and run `make proto`.
- `internal/apisrv/{admin,frontend,auth}` — gRPC service implementations (business logic entry points).
- `internal/store` — MySQL repository; sub-packages per domain. Interfaces live in `internal/dependency` (mocks generated by mockery into `internal/dependency/mocks`).
- `internal/api/http` — server: gRPC + grpc-gateway REST + Swagger UI, health checks, webhooks.
- `internal/dto` ↔ `internal/entity` — transport vs domain models.
- Cross-cutting: `internal/{cache,middleware,ratelimit,circuitbreaker,currency,rates}`.

### HTTP surface (`:8081`)
- gRPC services + REST/JSON gateway for all three services, Swagger UI at `/`.
- Health: `/readyz` (DB ping), `/livez`.
- Webhooks: Resend bounce/complaint + list-unsubscribe (`mail.NewWebhookHandler`).

## Configuration

`config/cfg.go` loads config with **viper**: a TOML file (default search `./config/config.toml`, or `-c <path>`)
overlaid by **environment variables, which take precedence**. Each setting is bound in `bindEnvVars`
(e.g. `mysql.dsn` ← `MYSQL_DSN`, `auth.jwt_secret` ← `AUTH_JWT_SECRET`).

- DB connection: read from the `MYSQL_DSN` env var (or constructed from `db.*` / `MYSQL_*` parts). The
  `DATABASE_URL` DO binding is **not** parsed by the app.
- TLS to managed MySQL uses `tls=custom`, registered from a CA cert file (`MYSQL_TLS_CA_PATH`) or
  `db.CA_CERT`. The runtime image ships the cert at `/etc/grbpwr-products-manager/certs/ca-certificate.crt`.
- When adding a new config field: add it to the struct in `config/cfg.go` (+ the relevant package `Config`),
  add a `viper.BindEnv` line, and set it in `.do/app.yaml` / `.do/app-beta.yaml`.

## Database / migrations

MySQL via `sqlx`. Numbered SQL migrations in `internal/store/sql/` (`0001_…` … latest). Applied on boot when
`MYSQL_AUTOMIGRATE=true` (true on **both** beta and prod — a master merge deploys prod and auto-applies pending
migrations). Because of this, a migration that fails on prod-specific data halts prod startup: every migration
must be safe against existing data (e.g. dedupe before adding a UNIQUE index — see `0059`), and it reaches beta
first via the feature→beta→master flow. Add a new migration as the next-numbered `NNNN_description.sql` file; do
not edit already-applied migrations.

New migrations must be **idempotent** so a mid-file failure (MySQL DDL auto-commits, so a rollback can't undo it —
the schema is left half-applied with no `gorp_migrations` row, and the next boot re-runs the file from the top).
Use `CREATE TABLE IF NOT EXISTS` (and symmetric `IF EXISTS` on drops), and never `DROP CHECK` an auto-generated
`<table>_chk_<n>` name — those are positional and drift across schema history; name constraints explicitly and drop
them by that stable name. Prefer several small migrations over one large destructive overhaul (see `0079`). The
lint in `internal/store/migrationlint` enforces this for migrations numbered above its grandfathered baseline.

Numbers must be **unique**. Historical exception: two files share prefix `0003`
(`0003_add_announce_translations.sql` and `0003_add_product_version.sql`) — both are already applied on prod, so
do **not** rename them (sql-migrate tracks by full filename; renaming reapplies). Always take the next free unique
number for new migrations; the `migrationlint` numbering test fails on any other collision.

## Deployment & environments

Two DigitalOcean App Platform apps, configured **purely via env vars** (no code branching):

- **prod**: app `grbpwr-backend`, branch `master`, `backend.grbpwr.com`, DB `grbpwr`, spec `.do/app.yaml`.
- **beta**: app `grbpwr-backend-beta`, branch `beta`, `backend-beta.grbpwr.com`, DB `grbpwr_beta` (same
  cluster), `MYSQL_AUTOMIGRATE=true`, Stripe TEST keys, analytics off, spec `.do/app-beta.yaml` (gitignored —
  holds plaintext secrets until DO encrypts them).

Branch flow: feature → `beta` (auto-deploys beta) → verify → merge to `master` (deploys prod). Frontends
(`grbpwr.com` / `admin.grbpwr.com` and their `beta.` / `admin.beta.` counterparts) are separate Vercel
projects, not in this repo. See `git log` and project memory for the full beta bring-up details.

## Two EUR figures on an order (do not confuse them)

`customer_order` carries two independently-derived EUR amounts — pick the right one:

- **`total_price_eur`** — a *loyalty* qualifying-spend snapshot, computed at order time by
  reconstructing from EUR catalogue prices (`internal/store/order/eur_snapshot.go`). It exists to
  accrue tiers immediately, for every payment method. **Never use it in revenue/margin metrics.**
- **`total_settled_base`** — the *actual* Stripe settlement in EUR at the payment FX rate (captured
  in `UpdateSettledBaseAndFee`). This is the **authoritative revenue figure** metrics must read.

They diverge (FX spread, rounding, promos). At capture `UpdateSettledBaseAndFee` collapses the
loyalty snapshot onto the settled fact when they agree within `settledEURReconcileTolerance` (2%);
a larger gap is left untouched and logged (rewriting qualifying spend would silently move tiers).
Historical rows are never revisited.

## Material warehouse vs sales COGS (do not cross the streams)

The new-flow material warehouse (`material_stock` / `material_stock_movement`, NF-01…NF-09) makes
production cost more *accurate* — a run's actual cost is derived from the materials issued to it, and
`product.cost_price` is set from that actual at receive. But it does **not** change how revenue or
sales margin is computed. The sales-margin chain is unchanged and one-directional:

material issues → production-run actual cost → `product.cost_price` at receive (provenance
`production_run`) → snapshot into `order_item` at sale → COGS in metrics.

So **sales/margin metrics read the order-time cost snapshot, never the live warehouse.** The warehouse
feeds *inventory* views only — `GetInventoryValuation` (raw/WIP/write-off money), the
`low_material_stock` / `stale_open_production_run` dashboard alerts, and the informational
`materials_from_stock_base` / `samples_cost_base` on `GetStyleEconomics` (which are **not** folded into
`net_after_dev`). Reading `material_stock` inside a revenue or gross-margin query is a bug: it would
double-count against the cost already snapshotted on the sold line and make margins depend on today's
stock rather than what the item actually cost when sold.

## Conventions

- Structured logging via `log/slog` (JSON handler); pass `ctx` and use `slog.String("err", err.Error())`.
- Wrap errors with `fmt.Errorf("...: %w", err)`.
- New external dependencies go behind an interface in `internal/dependency` with a generated mock.
- Don't commit secrets. `.env`, `config/config.toml`, `.do/app-beta.yaml`, and `*.crt`/`*.pem`/`*.key` are
  gitignored (the public DO DB CA cert at `config/certs/ca-certificate.crt` is the one allowed exception).
