## 1. Owner-reported issues → root cause → resolution

| # | Reported | Root cause | Resolution |
|---|---|---|---|
| R-1 | "owed to you" always **Failed to load** (`can't get receivables`, code 13) | sqlx's named-param scanner reads `':'` inside SQL string literals and `--` comments as an empty bind name. `GetReceivables` had BOTH (`SUBSTRING_INDEX(…, ':', 1)` + a comment with `bank:<id>`), so every call failed. `GetPayables` has no colon — that's why it worked. Same trap as tech-card `afbdcf0` and JPK `dfb69b4`. | **Fixed.** Separator → `CHAR(58)`, comment → Go side; systemic: `storeutil.makeQuery` now skips sqlx entirely for parameterless queries + regression test. |
| R-2 | Cash flow: no depreciation add-back / no operating breakdown / no drill-down | Backend already produced the full indirect-method statement (net profit, `Add back: depreciation (6370)`, ΔAR/ΔInventory/ΔPrepaid/ΔAP/ΔAccrued/ΔIncomeTax/ΔVAT/ΔPrepayments — `internal/accounting/reports.go:84`). The FRONTEND rendered only the three subtotals as a waterfall and threw the lines away. | **Fixed (FE).** cash-flow.tsx renders every line per section; account codes parsed from labels become drill-chips → ledger (same range), net-profit line links to P&L. ## 2. New findings from this audit (not previously known)

| ID | Sev | Finding | Status |
|---|---|---|---|
| F-1 | **HIGH** | **Period close had NO gate on unmatched bank lines.** A month could close with real bank money absent from the ledger — the Dr==Cr gate can't see lines that were never posted. | **Fixed**: gate 3d in `ClosePeriod` (`periods.go`) — unmatched lines booked in the month → `ErrAcctPeriodNotReady`. |
| F-2 | MED | **Corp-tax "reverse then re-accrue" was broken** (13a): reversed accrual kept matching `postedCorpTax` (stale figure reported as accrued) and the re-post collided with the unique key (no new entry). | **Fixed**: `postedCorpTax` excludes reversed + matches versions; new accruals use opex-style versioned keys `corp_tax:<from>:<to>[:vN]`. |
| F-3 | MED | **Double-expense trap:** opex/shipping are accrued Dr 6xxx / Cr 2030 by the worker; paying the bill from the bank inbox against a 6xxx account books the expense twice. Settlement of an accrual = 2030 (receipts = 2010). | **Mitigated (FE)**: bank post modal warns on any 6xxx counter-account. Consider a bank-rule lint later. |
| F-4 | MED | **Recon bank block was blind to IGNORED lines** — a real expense wrongly ignored vanished from every report. | **Fixed**: ignored-in-period lines now listed in the block ("deliberately not booked; check…"); TotalCount stays actionable-only. Longer-term: ingest a period-end statement balance to assert a real 1010 delta. |
| F-5 | LOW/MED | **FX fold uses TODAY's rate**, not `occurred_at` (documented phase-1 simplification). Backdated non-EUR manual/bank entries fold at the wrong economic rate; Dr==Cr unaffected. | **Accepted for now**; follow-up: thread `occurred_at` into `GetCostingFxRatesToBase`. |
| F-6 | LOW | AP/AR "overdue" can't be computed — no invoice due dates exist anywhere in the model. The tab dot uses open-balance + anomaly semantics instead. | Follow-up: `due_date` on supplier receipts (and bank-invoice orders) if aging matters. |
| F-7 | LOW | `IgnoreBankTxnRequest.reason` is accepted by the API and **discarded** (no column). | Follow-up: `0202_bank_txn_ignore_reason.sql` + persist + surface on ignored rows. |
| F-8 | LOW | Depreciation month-entries use non-versioned keys (`asset:<id>:<YYYY-MM>`) — a reversed month can't be re-posted (same shape as F-2). Disposal posts no derecognition entry (residual NBV stays on 1220/1225). | Accepted (rare); version keys if the workflow appears. |
| F-9 | LOW | Timezone edges: `firstOfDayUTC` (A-6) pins occurred_at to UTC dates and sessions don't pin `time_zone` (C-6). Consistent-UTC end to end; worst case a reversal near midnight CET lands a day off (and is period-guarded anyway). | **Accepted**: whole ledger is UTC-consistent; revisit only if local-date semantics are ever introduced. |
| F-10 | INFO | Revolut parser assumes `Amount` is gross-of-fee (fee booked separately to 6060 — MED-3 fix). Standard for Revolut business CSVs; a format change would double-count fees. | Follow-up: parser doc-test asserting the assumption. |
| F-11 | INFO | Verified OK during audit: cash-flow signs incl. 2080-in-VatDelta and the 1225/6370 exact cancellation; zero/negative-stock issues blocked (`ErrInsufficientMaterialStock` / `ErrSkipUncosted`); S2 refund restock (1130/5050); shipping income-vs-cost disjoint; delivered chain drains 2090/1140 to zero; depreciation/corp-tax rounding; no remaining `':'` hazards in params-carrying accounting SQL. | — |
