## Процесс (отработан фазой 1 — повторяем на каждую волну)

1. Дип-верификация якорей волны против кода (файлы:строки, сигнатуры) + уточнение спеки →
   правки волнового файла.
2. Дип-прогон «вероятные ошибки имплементации» + корнер-кейсы.
3. Агенты (Opus — деньги/логика, Sonnet — механика/UI) по нарезке волнового файла;
   интеграторский свип; аудит целостности скриптом.
4. Локально: make build + тесты + бета-деплой волны; смоук из DoD волны.

Правила фазы 1 действуют: миграции идемпотентны с именованными CHECK (номера от текущего
max+1 — проверять перед каждой волной!), леджер append-only/сторно, «правило локов» воркера,
uncosted не выдумываем, деньги decimal, RBAC на каждый новый RPC, контракт через зеркало
grbpwr-proto (байт-в-байт + mirror-git-ref), UI — существующий кит.

## Изменение учётной политики (волна 2 — фиксируем заранее)

Переход на delivered-признание — **cutover датой**, не ретроактивно: заказы, оплаченные до
`accounting.delivered_recognition_from`, живут по старой политике (S1 при confirmed);
после — новая (prepayment при оплате, revenue при delivered). Обе ветки сосуществуют в
воркере постоянно (история не переписывается). VAT-момент остаётся при оплате (Cr 2070 из
prepayment-проводки) — налоговая точка supply ≠ бухгалтерское признание, это допустимо и
сильно проще; подтвердить у бухгалтера при старте волны 2.

## Осознанно ВНЕ фазы 2 (чтобы не всплыло как «упустили»)

- **KSeF-интеграция** (обязательный PL e-invoicing B2B): отдельный проект с внешним API и
  сертификатами; волна 1 готовит данные (B2B-инвойсы с VAT-номерами), но не подключает KSeF.
- Автоматический банк-фид (API Revolut) — фаза 3; сейчас CSV.
- BTC/ETH-учёт (FIFO cost-basis, impairment) — крипта ограничена USDT по решению владельца.
- Мультиюрлицо/консолидация; бюджеты/план-факт; аудит-трейл изменений сверх сторно.
- Прод-переключение волн — тем же beta→soak→master циклом, по волне за раз.
