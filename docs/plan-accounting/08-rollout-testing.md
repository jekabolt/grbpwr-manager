# Часть 8: порядок работ, тестирование, rollout, риски, фаза 2

## Порядок PR (feature → `beta` → verify → `master`)

Каждый PR компилируется, проходит `make lint` + тесты, и деплоится на бету независимо.

| # | PR | Содержимое | Риск для прода |
|---|---|---|---|
| 1 | schema+entity | 0189, 0190, `entity/accounting.go`, enum-guard'ы в migrationlint | нулевой (новые таблицы) |
| 2 | store | `internal/store/accounting` (без reports), `dependency.Accounting`, wiring в `store.go`, моки, integration-тесты | нулевой (мертвый код) |
| 3 | rules | `internal/accounting` билдеры + юнит-тесты (чистая логика) | нулевой |
| 4 | producers | outbox-вставки в `OrderPaymentDone` / `RefundOrder` / `CreateCustomOrder` + тесты | низкий: +1 INSERT в трёх Tx; ревьюить как изменение order-флоу |
| 5 | worker | `internal/acctposting`, конфиг, app wiring, `.do/*.yaml` env | нулевой при `ACCOUNTING_ENABLED=false` |
| 6 | api | proto + dto + apisrv + RBAC + swagger | низкий (новые RPC) |
| 7 | reports | `reports.go`, `reconcile.go`, отчётные RPC | нулевой |
| 8 | alerts | dashboard-коды `acct_*` | нулевой |

Порядок важен: producers (4) можно мерджить до включения воркера — события копятся, воркер
потом дожуёт (идемпотентно). Это же даёт «тихий» запуск на проде: миграции + продьюсеры едут с
обычным релизом, воркер включается env-переключателем без деплоя кода.

## Чек-лист тестов (сводно; детали в частях)

- `migrationlint`: номера 0189/0190 уникальны, идемпотентность, именованные CHECK'и,
  enum-пары (`chk_acct_*` ↔ entity-константы).
- store integration (`*_integration_test.go`, реальная MySQL): баланс/идемпотентность/периоды/
  сторно (02), постинг всех источников + повторный прогон без дублей (07), отчёты на сквозном
  примере из 04 (06).
- юнит: билдеры проводок, пропорции VAT/shipping, caveat'ы (07).
- RBAC completeness-тест — упадёт сам, пока новые RPC не замаплены (05).
- `make generate` — моки перегенерены, `make build` зелёный.
- Ручной smoke на бете: тестовый заказ картой (Stripe TEST на бете) → confirmed → в течение
  минуты появляется проводка S1; рефанд → S2; приход материала → M1; receive руна → P1;
  ручная проводка через Swagger UI; TB сбалансирован; сверка revenue == 0.

## Rollout

1. Мердж PR 1–8 в `beta` → авто-деплой, `MYSQL_AUTOMIGRATE=true` применит 0189/0190 на
   `grbpwr_beta`. `ACCOUNTING_ENABLED=true`, `ACCOUNTING_START_DATE=<дата деплоя беты>`.
2. Неделя soak на бете: гонять заказы/рефанды/склад, смотреть `/statusz` (acctposting), сверку,
   алерты. Проверить закрытие первого месяца (ClosePeriod).
3. Merge в `master`. Первый прод-деплой: миграции применятся, `ACCOUNTING_ENABLED=false`.
   Убедиться, что события копятся (продьюсеры активны), затем выставить
   `ACCOUNTING_ENABLED=true` + `ACCOUNTING_START_DATE=<прод cutover>` в `.do/app.yaml` — с этой
   даты леджер прода начинает жить. Занести стартовые остатки (если решат) ручной проводкой.
4. Первые 2–4 недели: раз в неделю смотреть `GetAcctReconciliation` перед тем, как доверять
   P&L. Закрытие месяца — после появления привычки.

Ежемесячная рутина (числа 3–5-го, владелец — тот, кто ведёт финансы):

1. `GetAcctReconciliation` за месяц — разобрать все Pending (`manual entry required`,
   зависшие settled) и внести недостающие ручные проводки (payout'ы, оплаты поставщикам,
   VAT, зарплаты, chargebacks).
2. Сверить сальдо 1030 с реальным балансом Stripe Dashboard, 1010 — с банковской выпиской
   (руками; автоматизация — фаза 2).
3. `CloseAcctPeriod` — если не закрывается, ответ перечислит, что не готово.
4. Снять P&L/BS месяца — они теперь иммутабельны.

Откат: `ACCOUNTING_ENABLED=false` останавливает постинг; продьюсеры безвредны (копят строки);
таблицы изолированы — дропать ничего не нужно.

## Риски и осознанные допущения

| Риск/допущение | Митигация |
|---|---|
| Пропорция `k = settled/total_price` для VAT/shipping — приближение (Stripe FX применяется к сумме целиком, это точно; но vat_amount — снапшот расчётный) | дельты видны в сверке; VAT-фаза 2 пересчитает по режимам |
| `payment_fee` не уменьшается при рефанде → 6050 слегка завышен относительно «чистой» комиссии | задокументировано; Stripe так и берёт — это факт, не ошибка |
| Заказы non-EUR non-Stripe (USDT-кастомы) не постятся автоматически | алерт `acct_manual_entry_required`; ручная проводка |
| OPEX-правки закрытого месяца не перепощиваются | warning + reopen периода руками; политика «закрыл — не трогай» |
| 2010 AP — агрегат без субледжера поставщиков | приемлемо для масштаба; AP-субледжер в фазе 2 |
| FG в леджере ≠ FG в `GetInventoryValuation` (живой cost_price, ручные стоки) | это два разных вопроса («что стоило» vs «что стоит сейчас»); сверка объясняет дельту — то же разделение, что в CLAUDE.md про warehouse vs COGS |
| Uncosted-движения не постятся → баланс 1110 занижен | сверка перечисляет; дисциплина costed-приходов уже культивируется складом |
| P&L без расходов доставки (4110 есть, 6030 нет) — маржа завышена | постоянный caveat в P&L (06); `shipment.actual_cost` → фаза 2 |
| P&L до налога на прибыль | постоянный caveat; начисление CT — ручная проводка бухгалтера или фаза 2 |
| Stripe chargeback/dispute не порождает события (RefundOrder не вызывается) | деньги уходят мимо леджера → расхождение 1030 при ручной сверке со Stripe; ручная MN-проводка (04); webhook `charge.dispute.*` — фаза 2 |
| Продьюсер добавляет INSERT в Tx оплаты | таблица локальная, без внешних вызовов; ревью PR 4 отдельно; откат = revert одного маленького диффа |
| Двойной запуск воркера при будущем масштабировании | сейчас 1 реплика; при изменении — `GET_LOCK` |

## Фаза 2 (бэклог, в порядке ценности)

1. **VAT-движок по режимам** (лист VAT&Crypto): классификация заказа (страна доставки,
   склад отгрузки, B2C/B2B) → OSS/WNT/WDT/export; счета 2080 VAT Input, 4310 B2B; отчёты для
   JPK_VAT/OSS. База уже есть: `address.country_code`, `vat_rate`, `vat_amount`.
2. **Cash Flow statement** (indirect) — чистая производная от леджера: NP + ΔBS-статей.
3. **Financial Health ratios** поверх леджера (часть уже есть в `GetMetrics` — свести в одну
   секцию, добавить current/quick ratio, GMROI по-леджерному).
4. **Крипто-учёт FRS 105**: счёт 1050 Crypto Wallet, 7010/7020/7030; cost-basis по выбытиям;
   `USDT` уже легален как expense-валюта, курсы — `costing_fx_rate`.
5. **GBP-отчётность для filing**: пересчёт P&L/BS в GBP по курсам на дату/среднему за период
   (`costing_fx_rate` уже хранит историю) + FRS 105 форматы (7 строк IS, Format 1 BS).
6. **AP/AR-субледжеры**: справочник поставщиков, привязка платежей к приходам; AR для
   bank-invoice кастомов.
7. **Банковский CSV-импорт** в инбокс «Unsorted» с полуавтоматическим маппингом на счета.
8. **Dev-expenses и фактическая доставка** (`tech_card_dev_expense` → 6210/R&D;
   `shipment.actual_cost` → 6030) как pull-источники.
9. **Discounts в P&L**: строка «Less: Discounts» из `promo_discount_pct`-аналитики (счёт 4030).
10. **Начисление налога на прибыль** (2050 Income Tax Payable + строка Net Profit after tax).
11. **Stripe disputes**: webhook `charge.dispute.created/closed` → автопроводка chargeback'а.
12. **Точка признания по доставке** (delivered-based recognition + 1140 Inventory in Transit),
    если потребует бухгалтер, — смена продьюсера события без изменения правил.

## Definition of Done фазы 1

- Каждая оплата/рефанд/движение склада/receive/OPEX-месяц отражены проводкой либо явно видны в
  сверке как пропущенные с причиной.
- TB всегда сбалансирован; BS CHK == 0; P&L revenue сходится со сверкой в пределах явных дельт.
- Ручная проводка и сторно работают из админки; период закрывается и защищает журнал.
- Прод живёт с включённым воркером ≥ 2 недель без `acct_*`-алертов уровня error.
